What is a Data Schema?

Definition

A data schema is a blueprint that defines the structure, organization, and constraints of your dataset.

Think of it as the architectural plan for your data. Just as a building needs blueprints before construction, your dataset needs a schema before collection.

定義

データスキーマは、データセットの構造、組織、および制約を定義する設計図です。

データの建築計画として考えてください。建物が建設前に設計図を必要とするように、データセットは収集前にスキーマを必要とします。

Key Components of a Data Schema

1. Field Names / フィールド名

Field names identify what information each column contains.

Examples:

  • company_name
  • industry
  • employee_count
  • founding_year

Best Practices: Use clear, descriptive names. Be consistent (lowercase, underscores instead of spaces).

フィールド名は、各列にどのような情報が含まれているかを識別します。

例:

  • company_name（企業名）
  • industry（業界）
  • employee_count（従業員数）
  • founding_year（設立年）

ベストプラクティス: 明確で説明的な名前を使用。一貫性を保つ（小文字、スペースの代わりにアンダースコア）。

2. Data Types / データ型

TEXT / テキスト: - Company names, descriptions, addresses - 企業名、説明、住所 NUMBER / 数値: - Employee counts, revenue, founding year - 従業員数、収益、設立年 CATEGORICAL / カテゴリカル: - Industry (Technology, Healthcare, Finance) - Company size (Small, Medium, Large) - 業界、企業規模 DATE / 日付: - Founding date, data collection date - 設立日、データ収集日 URL: - Company website links - 企業ウェブサイトリンク

3. Requirements / 要件

Required Fields: Must be present for every entry

  • Company Name - Essential identifier
  • Industry - Critical for classification
  • Description - Provides context

Optional Fields: Nice to have, but can be missing

  • Revenue - Often not publicly available
  • Employee Count - May be estimates
  • Culture Keywords - Subjective data

必須フィールド: すべてのエントリに存在する必要がある

  • 企業名 - 本質的な識別子
  • 業界 - 分類に重要
  • 説明 - コンテキストを提供

オプションフィールド: あると良いが、欠損していても可

  • 収益 - 多くの場合公開されていない
  • 従業員数 - 推定値の可能性
  • 文化キーワード - 主観的データ

Designing an Effective Schema

Think About Your End Goal

Question: What do you want to do with this data?

質問: このデータで何をしたいですか？

For our project: Build a recommendation system that suggests companies based on personal career interests.

私たちのプロジェクトでは: 個人のキャリア興味に基づいて企業を提案する推薦システムを構築します。

What Information Helps You Decide?

Think about what influences your interest in a company:

  • Industry: Do you prefer tech, healthcare, or finance?
  • Size: Startup or established corporation?
  • Location: Prefer local or willing to relocate?
  • Mission: Does their description align with your values?
  • Growth: Young company or long-established?

どの情報が決定に役立ちますか？

企業への興味に影響を与えるものについて考えてください：

  • 業界: テック、ヘルスケア、金融のどれを好むか？
  • 規模: スタートアップか確立された企業か？
  • 所在地: 地元を好むか、移転する意志があるか？
  • ミッション: 説明はあなたの価値観と一致するか？
  • 成長: 若い企業か長く確立された企業か？

Recommended Schema for Company Dataset

REQUIRED FIELDS (Must Collect) / 必須フィールド（必ず収集）: 1. company_name (text) - Full official company name - 正式な企業名の完全版 2. industry (categorical text) - Standardized industry category - 標準化された業界カテゴリ 3. description (text) - Brief company overview (2-3 sentences) - 簡単な企業概要（2-3文） RECOMMENDED FIELDS (High Priority) / 推奨フィールド（高優先度）: 4. company_size (categorical: Small/Medium/Large) - Based on employee count - 従業員数に基づく 5. location (text: "City, Country") - Primary headquarters location - 主要本社所在地 6. founding_year (number: YYYY) - Year company was established - 企業が設立された年 7. website (URL) - Official company website - 公式企業ウェブサイト OPTIONAL FIELDS (Nice to Have) / オプションフィールド（あると良い）: 8. employee_count (number) - Approximate number of employees - おおよその従業員数 9. revenue (number) - Annual revenue if available - 利用可能な場合の年間収益 10. culture_keywords (text) - Words describing company culture - 企業文化を説明する言葉

Data Collection Strategy

Step 1: Identify Data Sources

Primary Sources

Your main source for bulk data collection:

  • Job listing websites (approved in Week 4)
  • Company directory sites
  • Industry-specific databases

Goal: Collect basic information for 20-30 companies from one comprehensive source.

主要ソース

バルクデータ収集のためのメインソース：

  • 求人ウェブサイト（第4週で承認済み）
  • 企業ディレクトリサイト
  • 業界固有のデータベース

目標: 1つの包括的なソースから20-30社の基本情報を収集。

Secondary Sources

Use to fill gaps in your data:

  • Individual company websites
  • Wikipedia or business profile sites
  • LinkedIn company pages
  • News articles about companies

Goal: Supplement missing fields like founding year, employee count, or better descriptions.

二次ソース

データのギャップを埋めるために使用：

  • 個別企業ウェブサイト
  • Wikipediaまたはビジネスプロフィールサイト
  • LinkedIn企業ページ
  • 企業に関するニュース記事

目標: 設立年、従業員数、またはより良い説明などの欠損フィールドを補完。

Step 2: Map Fields to Sources

Before collecting, plan where each field will come from:

収集前に、各フィールドがどこから来るかを計画します：

Job Listing Sites Usually Provide: ✓ Company name ✓ Industry ✓ Basic description ✓ Location ✓ Sometimes company size 求人サイトは通常以下を提供： 企業名、業界、基本的な説明、所在地、時々企業規模 Company Websites Usually Provide: ✓ Detailed description ✓ Founding year ✓ Mission/culture information ✓ Accurate employee counts 企業ウェブサイトは通常以下を提供： 詳細な説明、設立年、ミッション/文化情報、正確な従業員数 LinkedIn/Wikipedia Usually Provide: ✓ Founding information ✓ Size estimates ✓ Industry classification ✓ Brief overviews LinkedIn/Wikipediaは通常以下を提供： 設立情報、規模推定、業界分類、簡単な概要

Step 3: Execute Collection Systematically

Collection Workflow / 収集ワークフロー:

  1. Start with primary source - Get bulk data (company names, industries, descriptions)
    主要ソースから始める - バルクデータを取得
  2. Export to CSV - Save your initial dataset
    CSVにエクスポート - 初期データセットを保存
  3. Identify gaps - Which fields are missing or incomplete?
    ギャップを特定 - どのフィールドが欠損または不完全か？
  4. Use secondary sources - Fill gaps systematically
    二次ソースを使用 - 体系的にギャップを埋める
  5. Standardize format - Apply consistent formatting rules
    フォーマットを標準化 - 一貫したフォーマットルールを適用

Data Quality Principles

The 4 C's of Data Quality

1. Completeness / 完全性

Definition: How much of your data is filled in vs missing?

Goal: Aim for 80%+ completeness on required fields

How to Measure:

  • Count non-empty cells for each field
  • Calculate: (filled cells / total cells) × 100%
  • Example: 25 out of 30 companies have industry → 83% complete

定義: データのどれだけが埋まっているか、対、欠損しているか？

目標: 必須フィールドで80%以上の完全性を目指す

測定方法:

  • 各フィールドの空でないセルをカウント
  • 計算：（埋まったセル / 総セル）× 100%
  • 例：30社中25社が業界を持つ → 83%完全

2. Consistency / 一貫性

Definition: Are you using the same formats and categories throughout?

Examples of Inconsistency:

  • Industry: "Tech", "IT", "Technology", "Information Technology" (should all be "Technology")
  • Location: "Tokyo", "Tokyo, Japan", "Tokyo JP" (should all be "Tokyo, Japan")
  • Company names: "Sony Corp", "Sony Corporation", "SONY" (choose one standard)

定義: 全体で同じフォーマットとカテゴリを使用していますか？

不整合の例:

  • 業界：「Tech」、「IT」、「Technology」、「Information Technology」（すべて「Technology」であるべき）
  • 所在地：「Tokyo」、「Tokyo, Japan」、「Tokyo JP」（すべて「Tokyo, Japan」であるべき）
  • 企業名：「Sony Corp」、「Sony Corporation」、「SONY」（1つの標準を選択）

3. Correctness / 正確性

Definition: Is your data factually accurate?

How to Verify:

  • Cross-reference with official company websites
  • Check for obvious errors (founding year in future, negative employee counts)
  • Validate URLs actually work
  • Look for outdated information

定義: データは事実として正確ですか？

検証方法:

  • 公式企業ウェブサイトと相互参照
  • 明らかなエラーをチェック（未来の設立年、負の従業員数）
  • URLが実際に機能することを検証
  • 古い情報を探す

4. Clarity / 明確性

Definition: Is your data understandable and well-organized?

Best Practices:

  • Use descriptive field names
  • Avoid abbreviations unless standard (URL is OK, "emp_cnt" is not)
  • Include metadata documenting your choices
  • Write clear descriptions of what each field means

定義: データは理解可能で適切に整理されていますか？

ベストプラクティス:

  • 説明的なフィールド名を使用
  • 標準でない限り略語を避ける（URLは可、「emp_cnt」は不可）
  • 選択を文書化するメタデータを含める
  • 各フィールドが何を意味するかの明確な説明を書く

Common Challenges and Solutions

Challenge 1: Can't Find Enough Companies

Problem: Only found 10-15 companies, need 20-30

問題: 10-15社しか見つからず、20-30社が必要

Solutions:

  • Combine multiple approved sources
  • Use different search terms or categories
  • Expand geographic scope
  • Include companies from different industries

解決策: 複数の承認されたソースを組み合わせる、異なる検索用語やカテゴリを使用、地理的範囲を拡大、異なる業界の企業を含める

Challenge 2: Missing Data Fields

Problem: Many companies missing founding year or employee count

問題: 多くの企業で設立年や従業員数が欠落

Solutions:

  • Focus on required fields first (name, industry, description)
  • Use secondary sources to fill critical gaps
  • Mark uncertain data with notes
  • Accept that optional fields may be incomplete

解決策: まず必須フィールドに焦点を当てる、二次ソースを使用して重要なギャップを埋める、不確実なデータにメモを付ける、オプションフィールドは不完全でも受け入れる

Challenge 3: Inconsistent Formatting

Problem: Industry categories vary wildly ("Tech", "IT", "Software", "Technology")

問題: 業界カテゴリが大きく異なる

Solutions:

  • Create standardization rules early
  • Choose 5-10 standard industry categories
  • Map variations to standards (all tech-related → "Technology")
  • Apply consistently to all entries

解決策: 早期に標準化ルールを作成、5-10の標準業界カテゴリを選択、バリエーションを標準にマッピング、すべてのエントリに一貫して適用

Documentation Best Practices

What to Document:

  • Schema: List of all fields with descriptions and data types
  • Sources: Which websites you used, with URLs
  • Collection Date: When you collected the data
  • Methodology: How you collected and standardized
  • Quality Metrics: Completeness percentages
  • Known Limitations: What's missing or uncertain

文書化すべき内容:

  • スキーマ: 説明とデータ型を含むすべてのフィールドのリスト
  • ソース: 使用したウェブサイト、URLあり
  • 収集日: データを収集した日時
  • 方法論: どのように収集して標準化したか
  • 品質指標: 完全性パーセンテージ
  • 既知の制限: 何が欠けているか、または不確実か

The quality of your dataset directly impacts ML model performance. A well-structured, complete dataset leads to better recommendations.

品質が重要！ データセットの品質はMLモデルのパフォーマンスに直接影響します。適切に構造化された完全なデータセットは、より良い推薦につながります。

Key Takeaways

Remember / 覚えておいてください

  • Focus on required fields first, then enhance
  • Consistency is more important than perfection
  • Document everything for future reference
  • Quality data = Better ML performance

良いスキーマが効果的なデータ収集をガイド、収集前に計画する、まず必須フィールドに焦点を当てる、一貫性は完璧さより重要、将来の参照のためにすべてを文書化、品質データ = より良いMLパフォーマンス
