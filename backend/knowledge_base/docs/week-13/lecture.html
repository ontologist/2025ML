STEP 2: System Refinement and Improvement

Optional Refinements Based on Evaluation

Based on your Week 12 evaluation insights, you may choose to implement targeted improvements. Remember: perfection is not the goal—learning is!

第12週の評価の洞察に基づいて、ターゲットを絞った改善を実装することを選択できます。覚えておいてください：完璧が目標ではありません—学習が目標です！

Improvement Strategies

Choose 1-2 most impactful:

  1. Expand Dataset: Add 2-5 more companies for better training
  2. Enhance Features: Add new keywords based on what model missed
  3. Refine Engineering: Adjust features based on importance analysis
  4. Retrain Model: Train with improvements and compare results

Document Everything: Record what you changed, why, and the impact on performance.

改善戦略

最も影響力のある1-2を選択:

  1. データセット拡張: より良い訓練のために2-5社を追加
  2. 特徴量強化: モデルが見逃したものに基づいて新しいキーワードを追加
  3. エンジニアリング改良: 重要性分析に基づいて特徴量を調整
  4. モデル再訓練: 改善を伴って訓練し結果を比較

すべてを文書化: 何を変更したか、なぜ、そしてパフォーマンスへの影響を記録します。

Important Time Management Tip

Don't over-refine! Small improvements are perfectly acceptable. It's better to have complete documentation of your current system than incomplete documentation of an "improved" system.

過度に改良しないでください！ 小さな改善は完全に受け入れられます。「改良された」システムの不完全な文書よりも、現在のシステムの完全な文書を持つ方が良いです。

Example: Documenting Improvements / 改善の文書化の例 IMPROVEMENT LOG 改善ログ What Changed: Added 3 new companies to training dataset 変更内容：訓練データセットに3つの新しい企業を追加 Why: Model accuracy was 70%, wanted to provide more examples 理由：モデルの精度が70%だったので、より多くの例を提供したかった Impact: Retrained model accuracy improved to 75% 影響：再訓練されたモデルの精度が75%に改善 Insight: More training data helped, especially for edge cases 洞察：より多くの訓練データが役立ち、特にエッジケースに対して
STEP 3: Comprehensive Project Documentation

Documentation Requirements: 7 Essential Sections

Your project documentation is the complete story of your ML journey. It should be comprehensive, well-organized, and demonstrate your learning throughout the semester.

プロジェクト文書は、あなたのML旅の完全なストーリーです。包括的で、よく整理され、学期を通じてあなたの学習を実証する必要があります。

Section 1: Project Overview (1-2 pages)

What to Include:

  • Project Goal: Build ML system to recommend companies based on career interests
  • Motivation: Why this project matters to YOU personally
  • ML Workflow Summary: Overview of complete process
  • Personal Relevance: How you'll use this for job search
  • Key Results: What you accomplished

含めるべきもの:

  • プロジェクト目標: キャリアの興味に基づいて企業を推薦するMLシステムを構築
  • 動機: このプロジェクトがあなた個人にとってなぜ重要か
  • MLワークフロー要約: 完全なプロセスの概要
  • 個人的関連性: 就職活動にこれをどのように使用するか
  • 主要な結果: 達成したこと

Section 2: Data Collection Process (1-2 pages)

What to Include:

  • Data Sources: mynavi.jp and collection methods
  • Collection Methodology: How you scraped/gathered data
  • Ethical Considerations: Respecting robots.txt, rate limiting
  • Dataset Statistics: Number of companies, fields collected
  • Challenges Encountered: Issues and solutions
  • Screenshots: Evidence of data collection process

含めるべきもの:

  • データソース: mynavi.jpと収集方法
  • 収集方法論: データをどのようにスクレイピング/収集したか
  • 倫理的考慮事項: robots.txtの尊重、レート制限
  • データセット統計: 企業数、収集されたフィールド
  • 遭遇した課題: 問題と解決策
  • スクリーンショット: データ収集プロセスの証拠

Section 3: Data Preparation (2-3 pages)

What to Include:

  • Data Cleaning Operations: Standardization, validation, corrections
  • Missing Data Analysis: What was missing and why
  • Handling Strategies: Methods used for each type of missing data
  • Before/After Comparison: Data quality improvements
  • Validation: How you verified data quality
  • Code Snippets: Key cleaning operations

含めるべきもの:

  • データクリーニング操作: 標準化、検証、修正
  • 欠損データ分析: 何が欠けていて、なぜか
  • 処理戦略: 各タイプの欠損データに使用された方法
  • 前後比較: データ品質の改善
  • 検証: データ品質をどのように確認したか
  • コードスニペット: 主要なクリーニング操作

Section 4: Feature Engineering (1-2 pages)

What to Include:

  • Features Created: Text-based, numeric, categorical features
  • Rationale: Why each feature matters for recommendations
  • Keyword Selection: Career-related keywords chosen
  • Feature Importance: Which features were most predictive
  • Connection to Goal: How features help classify companies
  • Examples: Specific feature values for sample companies

含めるべきもの:

  • 作成された特徴量: テキストベース、数値、カテゴリカル特徴量
  • 根拠: 各特徴量が推薦にとってなぜ重要か
  • キーワード選択: 選択されたキャリア関連キーワード
  • 特徴量の重要性: どの特徴量が最も予測的だったか
  • 目標へのつながり: 特徴量が企業を分類するのにどのように役立つか
  • 例: サンプル企業の特定の特徴量値

Section 5: Model Training and Evaluation (2-3 pages)

What to Include:

  • Training Configuration: Model type, parameters, data split
  • Training Process: How model learned from labeled data
  • Performance Metrics: Accuracy, precision, recall, F1-score
  • Confusion Matrix: Where model succeeded and failed
  • Critical Analysis: What results mean in context
  • Limitations: Known weaknesses and constraints
  • Potential Improvements: What could make it better

含めるべきもの:

  • 訓練設定: モデルタイプ、パラメータ、データ分割
  • 訓練プロセス: モデルがラベル付けされたデータからどのように学習したか
  • パフォーマンスメトリクス: 精度、適合率、再現率、F1スコア
  • 混同行列: モデルがどこで成功し失敗したか
  • 批判的分析: コンテキストで結果が何を意味するか
  • 制限: 既知の弱点と制約
  • 潜在的改善: 何がそれをより良くできるか

要約、将来の応用、得られたML理解		 45 sec

Presentation Best Practices

Do's:

  • Tell YOUR story—make it personal
  • Use visuals: charts, screenshots, examples
  • Focus on the journey and learnings
  • Include 1-2 concrete examples of predictions
  • Practice with a timer—stay within 5 minutes
  • Prepare for questions about your process

すべきこと:

  • あなたのストーリーを伝える—それを個人的にする
  • ビジュアルを使用：チャート、スクリーンショット、例
  • 旅と学習に焦点を当てる
  • 1-2の具体的な予測の例を含める
  • タイマーで練習—5分以内に留まる
  • あなたのプロセスに関する質問に備える

Don'ts:

  • Don't overload slides with text
  • Don't focus only on technical details
  • Don't apologize for imperfect results
  • Don't read directly from slides
  • Don't go over time—practice pacing

すべきでないこと:

  • スライドをテキストで過負荷にしない
  • 技術的な詳細にのみ焦点を当てない
  • 不完全な結果を謝罪しない
  • スライドから直接読まない
  • 時間を超過しない—ペーシングを練習

import os import pandas as pd def validate_project_submission(project_folder): """ Validate ML-101 Final Project Submission ML-101最終プロジェクト提出の検証 """ print("=== ML-101 Project Validation ===\n") validation_results = { 'passed': [], 'failed': [] } #
